# DragonOS Nix Repro Debug

## 目标

把“描述 bug 现象”推进到“可重复、可观察、可解释的根因”。

这个 skill 适合主动调试链路：

1. 理解症状和期望语义。
2. 在 DragonOS 代码里定位候选路径。
3. 添加有目的的日志或轻量插桩。
4. 按仓库的 Nix 流程构建并启动 DragonOS。
5. 在系统内执行复现步骤。
6. 根据证据缩小范围，直到给出根因和修复方向。

## 使用边界

优先用于这些情况：

- 用户已经给出 DragonOS 中的具体异常现象，希望 AI 自己复现
- 需要修改内核、驱动、VFS、网络、调度、IPC、用户态 app 或 init 脚本后重新启动系统验证
- 需要通过日志、计数器、一次性调试代码来观察真实运行路径
- 需要把“构建成功但不知道为什么错”推进到“拿到证据链”

遇到这些情况时不要只停留在静态阅读代码，应主动进入“改代码 -> 构建 -> 启动 -> 测试 -> 归因”的闭环。

如果问题明显是 Heisenbug、时序敏感、加日志就改变现象，优先参考 `dragonos-atomic-snapshot-debug` 的低扰动方法，不要无脑堆高频日志。

## 前置信息

开始前先整理出最小问题陈述：

- 现象：实际行为是什么
- 预期：按 Linux 6.6 语义应该怎样
- 触发方式：哪个程序、系统调用、脚本、启动阶段或设备路径能复现
- 影响范围：稳定复现、概率复现，还是仅在特定输入/时序下出现

如果用户没有给全，就先从仓库和现有代码中补齐最小上下文，不要一上来就盲改。

## 权限闸门

如果你预判后续步骤可能需要 `sudo` 或 root 密码，必须在最开始就问用户，不能做到一半再问。

典型例子：

- 当前环境启动 QEMU/KVM 需要提权
- 需要挂载、loop 设备、网络 tap/bridge、系统级调试接口
- 需要写入当前工作区之外的受限路径

如果整个流程不需要提权，就直接继续，不要无谓打断用户。

## 工作流

### 1. 先建立复现假设

先回答这几个问题：

1. 最可能出问题的子系统是什么
2. 当前仓库里是否已经有现成的复现程序、单测、gVisor 测例或 init 路径
3. 这次需要改的是内核、用户态、还是两者都要动

优先查找这些位置：

- `kernel/`
- `user/apps/c_unitest`
- 用户给出的测试程序
- 与问题相关的 `user/apps/*`、`user/sysconfig/etc/init.d/*`

如果能把复现收敛为一个最小命令、一个最小测试程序或一次启动流程，就先做这件事。

### 2. 设计插桩，不要先刷屏

日志或插桩必须服务于一个明确假设，例如：

- 某个条件位是否被置上
- 某次唤醒是否真的发生
- 某个对象状态是否在进入 syscall 前后发生错误变化
- 某个分支是否提前返回、重试过多或吞掉错误码

优先添加这些证据点：

- 函数入口/出口和关键参数
- 状态转换边界
- 阻塞前后
- 唤醒、完成、入队、出队、提交、回收等事件边界
- Linux 语义容易被改坏的返回条件

避免：

- 大段无差别 `debug!`/`println!`
- 无法关联上下文的随机打印
- 用 workaround 掩盖问题

如果需要临时复现辅助代码，尽量把它隔离在最小范围，并在最终结论里说明哪些改动只是调试用途。

### 3. 按 Nix 流程构建 DragonOS

严格参考仓库文档 `docs/introduction/develop_nix.md`。

默认流程：

1. 进入 `nix develop` 环境
2. 在仓库根目录运行 `make kernel`
3. 如果修改了 rootfs、用户态程序或 init 逻辑，运行 `nix run .#rootfs-x86_64`
4. 运行 `nix run .#start-x86_64` 启动 DragonOS

补充要求：

- 每次关键修改后都重新构建，不要假设旧产物仍然有效
- 如果只改了内核，通常至少重新执行 `make kernel`
- 如果改了 `user/` 下会进入镜像的内容，必须重新构建 rootfs
- 启动后确认你看到的是最新实例，而不是旧的 QEMU 会话

推荐把构建和启动命令、时间点、对应代码版本记在调试记录里。

更多命令细节见 [references/nix-qemu-workflow.md](references/nix-qemu-workflow.md)。

### 4. 启动后主动执行复现

不要停在“系统能起来”。进入系统后要继续推动复现：

- 运行目标测试程序或最小复现命令
- 观察串口/控制台输出
- 记录最后一个成功事件和第一个异常证据

如果交互步骤太繁琐，可以考虑：

- 在用户态测试程序里固定输入
- 暂时调整 init/启动脚本，使系统启动后自动运行一次复现

前提是改动范围要小、可解释，并且不能把调试脚本伪装成正式修复。

### 5. 以证据更新假设

每轮调试至少回答：

1. 这轮日志证明了什么
2. 哪些猜测被证伪了
3. 下一轮最小改动应该放在哪里

有价值的中间结论包括：

- 失败发生在进入某子系统之前还是之后
- 错误码、状态位、计数器是否已偏离 Linux 语义
- 某个对象是否在错误的生命周期阶段被消费、释放或重复处理
- 某个等待条件是否永远不可能满足

如果日志已经说明问题在更高层语义边界，不要继续往更底层乱挖；此时应回到契约、返回值和状态机设计。

### 6. 必要时切换到更低扰动方案

如果出现这些信号：

- 加日志后现象消失
- 复现概率显著变化
- 调度/时序相关路径被打印严重干扰

就停止继续加高频日志，转用原子计数器、静态快照或 `dragonos-atomic-snapshot-debug` 的方法。

### 7. 最终输出要给出根因链路

最终结论至少包含：

- 复现条件
- 实际观察到的关键证据
- 被证伪的主要分支
- 最可能的根因
- 根因为什么违反 Linux 6.6 语义或 DragonOS 设计意图
- 建议修复点

如果你已经顺手完成修复，也要分清：

- 哪些改动是调试插桩
- 哪些改动是真正修复

## 输出格式

默认按这个结构汇报：

### Bug Summary

- 症状
- 预期语义
- 当前复现方式

### Reproduction

- 使用了哪些构建/启动命令
- 在系统内执行了什么测试
- 复现是否稳定

### Evidence

- 关键日志/插桩结论
- 被证伪的假设
- 当前最强证据链

### Root Cause

- 根因判断
- 对应代码路径
- 为什么这是根因而不是表象

### Next Change

- 建议修复方案，或下一轮最小改动

## 关键提醒

- 不要把“多打点日志”当成分析本身；日志只是取证手段
- 不要绕过问题；修复必须符合 Linux 6.6 语义
- 不要忘记确认当前运行的真的是最新构建
- 不要在需要 root 权限时沉默尝试；要在一开始就问用户
